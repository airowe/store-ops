# Animation plans

Generated by `improve-animations` at commit `08b6e81`.

## Context: this codebase has an animation *absence*, not an animation problem

Worth knowing before you pick up a plan, because it explains why the list is short:

- **The web app (`cloud/web`) has exactly ONE transition in its entire source** — `app.css:74`, on the app-card hover. No modals, dropdowns, toasts, or overlays animate, because none of them animate at all. There are no motion libraries (no Framer Motion, no Tailwind, no Radix).
- **The mobile app has no motion code.** Reanimated, Gesture Handler, and Skia are installed, but only as transitive dependencies of `react-native-graph`, which powers one component (`RankTrendChart`). Nothing calls `withTiming`, `useSharedValue`, or `Animated` directly.
- **No motion tokens existed** — `packages/tokens` shipped color, type, spacing, and radius, but zero easing or duration.

So the usual audit findings (`ease-in` on UI, `scale(0)`, `transition: all`, keyframes on interruptible UI, unaccelerated Framer shorthands) **did not fire — there was no motion to get wrong**. Two of the four plans below are corrective; two are additive.

## Execution order

**001 must go first.** It is pure plumbing (no visible change), but 002, 003, and 004 all consume `--ease-out` and the `--duration-*` scale. Running any of them before 001 produces CSS that silently falls back to **no transition at all** — it will look like the plan did nothing.

```
001 (tokens)
 ├── 002 (appcard hover)      ← highest value: fixes a real touch-device defect
 ├── 003 (approval gate)      ← highest feel payoff: the product's defining moment
 └── 004 (timeline stagger)   ← polish; do last, or skip if effort is scarce
```

002, 003, and 004 are independent of each other and can be done in any order (or in parallel) once 001 lands.

One shared file to watch: **002, 003, and 004 all append to the same `@media (prefers-reduced-motion: reduce)` block** at the end of `cloud/web/src/app.css`. Whichever runs first creates it; the others extend it. If you run them in parallel, expect a trivial merge conflict there.

## Why 001 is timed the way it is

`cloud/web/src/app.css:2` says: *"The full design system ports in with the route PRDs."* This CSS is explicitly a placeholder. Establishing the motion scale **now, before that port**, is the whole point — it's cheap today, and it's what stops five near-identical hand-typed cubic-beziers from landing with the design system. Do it before the PRDs, not after.

## Things deliberately NOT planned

- **Mobile press feedback** (`mobile/src/components/primitives.tsx:172`) uses `opacity: 0.85` rather than the playbook's `transform: scale(0.97)`. Left alone on purpose: opacity-dimming is the platform-native press convention on React Native and it is applied consistently. Changing it is a taste call, not a fix.
- **`RankTrendChart`'s feel** cannot be judged from source — its easing and spring live inside `react-native-graph`. Whether the scrub feels laggy needs a real device. Not guessed at.
- **The web's index-key bug** (`AppDetailView.tsx:93` — `key={\`${a.at}-${i}\`}` on a `.slice(-8)` moving window) is a genuine correctness defect, the same one already fixed on mobile in `mobile/src/lib/rankSeries.ts`. It is **not** an animation issue and needs its own change with its own tests — but plan 004 touches that exact code and calls it out, because an unstable key makes a stagger visibly misfire.
