## 1. Summary

ShipASO is an App Store Optimization (ASO) product for independent iOS and
Android developers. It audits a developer's live app-store listing, researches
the keywords that listing should target, optimizes the listing copy to each
store's character limits, hands the developer the exact commands (or a pull
request) to publish the change, and then reads the resulting search rank back
over time to prove the change worked.

ShipASO requests read-only access to the Google Ads API's **Keyword Planning
Services** for a single purpose: to enrich its keyword research with **real
historical search-volume metrics** for candidate keywords, replacing the
internal 0–100 volume *proxy* the product currently uses. We do not create,
manage, or modify any Google Ads campaigns, ad groups, ads, budgets, or bids. We
do not act on behalf of advertisers. We do not resell, redistribute, or expose
raw Google Ads data to end users.

## 2. Business model and customers

**What ShipASO is.** A hosted SaaS plus a free, open-source (MIT) plugin for the
Claude Code developer tool. The hosted agent runs the full ASO loop on a weekly
schedule; the free plugin lets a developer run the same loop themselves.

**Primary customers.** Independent and small-studio mobile app developers who
publish their own iOS / Android apps and want to improve their organic
app-store search rank without hiring an ASO consultant or paying for an
enterprise ASO suite.

**The value exchange.** Customers pay a subscription for the hosted agent. In
return ShipASO does the recurring ASO work for them — research, copy
optimization, a publish-ready change, and weekly proof that the rank moved —
and notifies them only when there is a real, approved-pending change to ship.
The free plugin is a lead-generation and trust-building channel; the paid tier
sells the convenience of not having to remember to run the loop.

**Who has access to the Ads API integration.** Internal only. The Google Ads
API is called server-side by ShipASO's backend using airowe, LLC's own developer
token and OAuth credentials. End users never receive a Google Ads token, never
authenticate against Google Ads, and never see raw Ads API responses. They see
only a derived keyword score in ShipASO's own UI.

## 3. Why the Google Ads API is necessary

ShipASO's keyword research today ranks candidate keywords with a composite score:

```
score = volume * 0.4  +  (100 − difficulty) * 0.3  +  relevance * 0.3
```

The `relevance` term is computed from the app's own listing, and `difficulty` is
derived from the live competitive field on the app stores. The `volume` term —
how much real search demand a keyword has — is currently an internal **0–100
proxy**, because ShipASO has no licensed source of true search-volume data.

**This is exactly the gap the Google Ads API fills.** A keyword with a great
relevance/difficulty profile but near-zero real search demand is a wasted
metadata slot; one with high demand is worth fighting for. Real historical
search-volume metrics let ShipASO rank candidate keywords by *actual* demand
instead of an estimate, which directly improves the single most important output
of the product: which keywords go into the customer's limited app-store metadata
fields.

**Specific API feature required:**

- `KeywordPlanIdeaService.GenerateKeywordHistoricalMetrics` — read-only. Given a
  short list of candidate keyword strings (already generated by ShipASO from the
  app's listing and competitive set), it returns historical monthly search
  volume and competition indices. ShipASO uses the **avg. monthly searches**
  field to replace its internal volume proxy.

**Features explicitly NOT used:** campaign creation/management, ad/ad-group
mutate operations, bidding, budgets, conversion tracking, remarketing audiences,
recommendations, or any `mutate` call of any kind. The integration is read-only
against a single service.

## 4. Data flow

```
App listing (iTunes / Google Play public data)
        │  ShipASO derives candidate keywords from the listing + competitors
        ▼
Candidate keyword strings  ──►  Google Ads API
                                GenerateKeywordHistoricalMetrics (read-only)
        ◄──  avg. monthly searches per keyword
        │  ShipASO maps volume → its 0–100 scale, feeds the keyword score
        ▼
Keyword score (volume·0.4 + (100−difficulty)·0.3 + relevance·0.3)
        │
        ▼
Optimized listing copy  ──►  developer approves  ──►  developer publishes
```

- The API is queried with **keyword strings only** — never user PII, never
  campaign data.
- Responses are used transiently to compute a score; ShipASO stores the derived
  0–100 score, not a verbatim copy of Ads API metrics held for resale.
- No Ads data is shown to end users in raw form and none is redistributed.

## 5. Access, security, and compliance

- **Token holder:** airowe, LLC, under MCC 650-477-9160. Calls are server-side
  with the LLC's own OAuth client; no third-party tool holds the token.
- **Internal users only:** employees/contractors of airowe, LLC. No customer-
  facing Ads API surface.
- **Read-only:** the integration calls a single read service and performs no
  `mutate` operations.
- **No resale / redistribution:** Ads data is consumed to compute a derived
  score and is not exposed, exported, or sold.
- **Rate / volume:** request volume scales with the number of ShipASO apps under
  management — a handful of candidate keywords per app per weekly run, well
  within standard quota.
- **Contact:** a domain email (e.g. `google-ads-api@shipaso.com`) is used as the
  developer contact, monitored for compliance correspondence.
